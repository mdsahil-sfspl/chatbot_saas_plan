Signup & onboarding — from admin-provisioned to self-serve

Today a person can only get in if an admin creates them. This task builds the product's front door: a public signup that spins up a new tenant, its first admin, and a working workspace — reusing the user-creation and auth plumbing you already have.

Where you are today

How people get in right now

gapNo self-serve signup — users are created by an admin via create_user and bulk-upload, or pushed by the master portal.
gapOne shared world — everything hangs off a single system_settings (used 97×). No tenant concept yet.
reuseAuth is ready — Argon2 hashing, sessions, and a must_change_pw first-login flow already exist.
reuseA max_users quota is already enforced — it just needs to become per-tenant.
reuseEmail service + UserCreate model exist — reuse them for verification and validation.
buildNew: a public /signup, a tenants record, default-workspace seeding, and a trial hook.
The sequencing insight that saves rework This task starts before full multi-tenant isolation (Task 12) lands. So build signup to create a tenant and stamp tenant_id from day one, even though row-level isolation is enforced later. That way Task 12 slots in cleanly and you never rewrite the signup path. Until isolation is live, keep public signups gated (invite-only or a staging flag).
The mental model

What one signup should do

A single self-serve signup turns into a whole ready workspace. Everything to the right happens in one flow.

What to do · in order

The four moves

Build the tenant-aware signup first, then make it safe, then make the workspace actually usable.

a

Create tenant + first admin in one transaction

~2 days
api/routers/signup.py (new)migrations/ (tenants)reuse: admin_router.create_user
  1. Add a tenants table (id, org name, plan, status, created_at). This is the same table Task 10 designs — build the minimal version now.
  2. Write the signup endpoint — one transaction so a half-made tenant can never exist. Reuse your existing password hashing and user-creation logic. 
    # api/routers/signup.py
@router.post("/signup")
async def signup(payload: SignupIn):
    async with db.transaction():
        tenant_id = await create_tenant(payload.org_name)
        await create_user(tenant_id, payload.email,
                          hash_password(payload.password), role="admin")
        await seed_defaults(tenant_id)          # move (c)
        await start_trial(tenant_id, "starter")  # move (d)
    await send_verification_email(payload.email)
    return {"tenant_id": tenant_id}
  3. Stamp tenant_id everywhere the new user's rows are written — so the account is isolation-ready before Task 12 turns on RLS.
b

Make it safe — verify, validate, rate-limit

~1.5 days
api/routers/signup.pyreuse: email_service · must_change_pw
  1. Email verification — reuse your email service to send a signed, expiring verification link; the workspace stays limited until it's confirmed.
  2. Password & input rules — enforce a minimum password policy in the SignupIn model; reject disposable domains if you want.
  3. Rate-limit the endpoint (ties to Task 2) so signup can't be scripted for abuse: 
    # signup is a prime abuse target — cap it per IP
@limiter.limit("5/hour", key_func=get_remote_address)
  4. Handle duplicates gracefully — a clear "email already registered" path, and re-send-verification instead of silent failure.
Depends on Task 5 (proxy fix)Rate-limiting signup by IP only works once the proxy trust is locked down — otherwise the IP can be forged. Make sure that fix lands alongside this.
c

Seed a workspace that actually works

~1.5 days
reuse: system_settings defaults · master-configcore/usage_limiter.py (per-tenant quota)
  1. Create a starter chatbot for the new tenant so the dashboard isn't empty — a scaffold they can immediately test.
  2. Copy default config per tenant — quotas (reuse the max_users / voice-quota logic, now scoped to the tenant), feature flags, and languages, seeded from your master-config defaults.
  3. Guide the first session — a short onboarding checklist (verify email → try the bot → upload a doc) that lands them on the dashboard.
d

Start the trial (billing hook)

~1 day
links to Task 13 (Razorpay)
  1. Put every new tenant on a trial plan at signup (status = trial, plan = starter), so feature flags and quotas apply from minute one.
  2. Leave a clean seam for Razorpay (Task 13) — signup creates the trial; billing later attaches the paid subscription and upgrades the plan/flags.
Order of workSignup can ship its trial state before Razorpay is wired. Don't block onboarding on billing — the two meet at the plan/flag layer.

Definition of done

The task is finished when…

Done when
  • A public /signup creates a tenant, its first admin user, and default config in one transaction (no half-made accounts).
  • Email verification works; a password policy is enforced; the endpoint is rate-limited per IP.
  • The new tenant lands in a working workspace — a starter chatbot, seeded quotas/flags, and a dashboard.
  • tenant_id is stamped on everything from day one, so Task 12's isolation drops in without rework.
  • Every new tenant starts on a trial plan, with a clean seam for Razorpay (Task 13) to attach billing later.
  • Until Task 12 isolation is live, public signup stays gated (invite or staging flag).
