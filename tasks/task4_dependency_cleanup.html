Task 4 — dependency cleanup & Docker slimming

Your image ships 282 pinned packages — including a whole web framework you never import, a plotting library you never call, and three copies of OpenCV for code that imports cv2 zero times. Here's exactly what to cut.

Where you are today

What the audit found

dupe3 OpenCV builds — contrib 4.10, python 4.8, headless 4.13. They collide in one cv2 namespace.
deadFlask + Werkzeug + flask-babel — a full web framework, never imported anywhere in your code.
deadmatplotlib — pinned, but never imported. Pure weight.
devDev/test tools in prod — pytest, mypy, bandit, pip_audit, schemathesis shipped in the runtime image.
riskypip install --no-deps — skips dependency resolution; one missing sub-dep = a runtime crash.
buildSingle-stage build — build-essential + dev headers stay in the shipped image.
goodBase image is python:3.11-slim-bookworm and a .dockerignore exists — solid starting point.
keepThe heavy OCR stack (paddleocr/paddlepaddle) is real — it's a feature, so keep it (or split it out later).
The tell that the list has drifted Your Python code imports cv2 zero times — OpenCV is only pulled in indirectly by PaddleOCR. Yet you pin three OpenCV builds by hand. That's the fingerprint of a pip freeze that grew over time and was frozen with --no-deps. Cleaning it up is safe, and it pays back in faster builds, smaller images, and fewer version clashes.
The mental model

Four buckets for every dependency

Sort each package into one of four buckets. Everything below the green line is bloat you can remove, move, or dedupe.

What to do · in order

The four moves

Start with the free wins (4a, 4b), then the structural fix (4c).

4a

Cut the dead weight & the OpenCV triplet

quick win · ~half a day
requirements.txtDockerfile (system libs)
  1. Delete the unused packages — verified absent from all imports: Flask, Werkzeug, flask-babel, matplotlib (double-check babel isn't used standalone before removing it).
  2. Collapse OpenCV to one headless build. The server has no GUI, and your code never imports cv2 directly: 
    # requirements.txt — remove these two:
opencv-contrib-python==4.10.0.84
opencv-python==4.8.1.78
# keep only:
opencv-python-headless==4.10.0.84
  3. Trim now-unneeded system libs in the Dockerfile — the headless build drops the GUI stack, so you can usually remove libgl1 libsm6 libxext6 libxrender-dev (keep ffmpeg, poppler-utils, libportaudio2).
4b

Split dev tools out of the prod image

quick win · ~2 hrs
requirements.txtrequirements-dev.txt (new)
  1. Create requirements-dev.txt and move the test/lint tools there: 
    # requirements-dev.txt — installed only in CI / locally
pytest==9.0.3
mypy==1.20.1
bandit==1.9.4
pip_audit==2.10.0
schemathesis==4.15.2
  2. Keep the prod image lean — it installs requirements.txt only. CI installs both.
  3. Wire the dev file into CI (Task 8) so linting/tests still run — just not inside the shipped image.
Why it matters beyond sizeTest/audit tooling in a production image is extra attack surface for zero runtime benefit. Shipping less code is shipping less risk.
4c

Fix the install strategy + go multi-stage

structural · ~1.5 days
Dockerfilerequirements.in → requirements.txt (lockfile)
  1. Replace --no-deps with a real lockfile. Keep a short requirements.in of the packages you actually depend on, and compile a fully-pinned lock (pip-tools or uv). Now transitive deps resolve correctly and you can install normally.
  2. Move compilers into a builder stage so they never ship in the final image: 
    # Dockerfile — multi-stage
FROM python:3.11-slim-bookworm AS builder
RUN apt-get update && apt-get install -y --no-install-recommends build-essential libpq-dev portaudio19-dev
COPY requirements.txt .
RUN pip wheel --wheel-dir /wheels -r requirements.txt

FROM python:3.11-slim-bookworm
RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq5 poppler-utils ffmpeg libportaudio2 curl && rm -rf /var/lib/apt/lists/*
COPY --from=builder /wheels /wheels
RUN pip install --no-cache-dir /wheels/*
COPY . .
  3. Rebuild and compare — note the image size before/after, and confirm the app boots and a smoke test passes.
4d

Optional: split the OCR stack into its own image

later · when it hurts
Dockerfile.ocr (new)
  1. PaddleOCR + paddlepaddle + paddlex are the single heaviest thing in the image (hundreds of MB). If OCR isn't on the hot path, move it to a separate worker image so the main API stays small and deploys fast.
  2. Only do this once the size actually hurts your deploys — don't add a second image before you feel the pain.

Definition of done

Task 4 is finished when…

Done when
  • 4a — a single opencv-python-headless remains; Flask/Werkzeug/flask-babel and matplotlib are gone; unused GUI system libs dropped from the Dockerfile.
  • 4b — dev/test tools live in requirements-dev.txt; the prod image installs runtime deps only.
