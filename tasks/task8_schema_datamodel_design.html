The picture in plain language
Task 11 was "walk each customer to the right house." Task 8 is drawing the house plan — and you're going to build this same house twenty times over.

So you settle three things now, while it's cheap: what rooms every house has (the tenant template), what stays in the shared office (billing, the client list, the phonebook), and how you renovate all twenty houses at once without one of them ending up with a different kitchen. Change the plan after twenty houses are built and every fix costs twenty times more.

1Decision 1 — Draw the boundary, and write it down

Your 46 tables already split cleanly. Task 8's job is to make that split explicit and enforced, so no future table lands on the wrong side by accident.

The rule to write down: a table goes in the central control DB if it describes a customer; it goes in the tenant DB if it describes a customer's content. Anything with a client_id column belongs to the centre — and, critically, nothing in a tenant database should ever have a client_id, because there's only one client in there.

Central control DB — one, shared (14 tables)

clients, client_servers, client_config, client_api_key_history, client_provision_log, billing, usage_logs, master_admins, master_sessions, master_settings, plan_defaults, llm_providers, voice_providers, sarvam_pricing

Tenant DB template — one copy per customer (~32 tables)

chatbots, chatbot_prompts, chatbot_usage_limits, the document tables, uploaded_files, conversation_details, conversation_summaries, conversation_analytics, conversation_feedback, user_credentials, password_reset_tokens, session_*, system_settings, audit_log, all admin_* tables, choose_pages

⚠️
The one genuinely ambiguous table — decide it explicitly. audit_log currently carries a client_id and logs in-tenant activity. Pick one: either it's a tenant table (drop client_id, each customer's audit trail lives in their own DB — better for DPDP data-residency and deletion requests), or it's central (keep it, you get one cross-customer view for support). My recommendation: tenant-side, with only provisioning/admin events mirrored centrally via client_provision_log, which you already have.

2Decision 2 — The biggest data-model problem in your schema

This is the finding that most justifies Task 8's 11 days. Right now, adding a chatbot requires running DDL.

Your schema has four near-identical document tables — and I mean identical:

-- it_documents_v2, hr_documents_v2, ops_documents, lavanya_documents
CREATE TABLE public.it_documents_v2 (
    id        bigint NOT NULL,
    content   text   NOT NULL,
    metadata  jsonb,
    embedding vectors.vector(3072)
);
-- ...and hr_documents_v2 is byte-for-byte the same shape. And ops. And lavanya.

-- plus a hand-written SQL function per table:
--   match_it_documents, match_it_documents_v2,
--   match_hr_documents_v2, match_lavanya_documents

And chatbots points at them dynamically:

CREATE TABLE public.chatbots (
    category        varchar(100) NOT NULL,   -- PRIMARY KEY (!)
    table_name      varchar(255),            -- "which table holds my docs"
    search_function varchar(255),            -- "which SQL function searches it"
    ...
);
🚩
Why this breaks multi-tenancy: to onboard a customer who wants a "Finance" bot, someone must CREATE TABLE finance_documents and CREATE FUNCTION match_finance_documents — in that customer's database. Self-serve onboarding becomes impossible, every tenant DB drifts to a different shape, and your app builds SQL from a string in table_name (an injection surface). Multiply that by 20 customers × 4 bots and no two databases will ever match.

The fix: one documents table, one search function

-- ONE table for all bots in a tenant DB
CREATE TABLE documents (
    id          bigserial PRIMARY KEY,
    chatbot_id  bigint NOT NULL REFERENCES chatbots(id) ON DELETE CASCADE,
    content     text   NOT NULL,
    metadata    jsonb  DEFAULT '{}'::jsonb,
    embedding   vector(3072),
    created_at  timestamptz DEFAULT now()
);
CREATE INDEX ON documents (chatbot_id);

-- ONE search function, parameterised by chatbot instead of by table name
CREATE FUNCTION match_documents(
    p_chatbot_id bigint,
    query_embedding vector,
    match_count int DEFAULT 8,
    min_similarity float DEFAULT 0.55
) RETURNS TABLE(id bigint, content text, similarity float) ...
✅
What this buys you: adding a chatbot becomes an INSERT, not a schema migration. Every tenant database has the same shape forever, so one migration script works everywhere. The table_name / search_function columns and the dynamic-SQL path disappear — and with them, an injection surface.
🔑
Also fix while you're here: chatbots's primary key is category (a varchar(100)). A human-editable label as a primary key means renaming a category breaks every foreign key pointing at it. Add a surrogate id bigserial PRIMARY KEY and keep category as a plain unique column.

3Decision 3 — The vector layer needs a decision now

Two things in your schema will bite hard once each customer has their own database.

🐌
There is no vector index anywhere in the dump. I searched all 3,181 lines: 55 indexes are defined, and not one is hnsw or ivfflat on an embedding column. Every similarity search is currently a brute-force scan of every row. That's survivable on one small dataset; across 20 customer databases it's the thing that makes the product feel slow — and no amount of prompt tuning fixes it.
📐
3072 dimensions is a costly default. Your embeddings are vector(3072) ≈ 12 KB per row before overhead. Task 8 should decide whether you actually need that: dropping to 768–1536 dims (or storing halfvec) typically cuts storage and search cost several-fold with little real accuracy loss for document Q&A — and standard pgvector HNSW indexes cap at 2000 dimensions, so 3072 constrains your indexing options. Decide the dimension once, here: it must be identical in every tenant DB, or search silently returns nonsense.
🔍
Worth confirming with Pritam: your dump uses CREATE EXTENSION vectors WITH SCHEMA vectors and types like vectors.vector(3072) — that's pgvecto.rs, not the standard pgvector extension. They are different projects with different index syntax and different operational maturity, and self-hosted customers will need whichever one you pick installed on their Postgres. Task 8 should make this an explicit, documented choice rather than an inherited accident.

You already have client_provision_log with action, status, payload, error, duration_ms — a well-designed audit trail for exactly this. Task 8's job is to define which steps get logged and what happens on partial failure (a database created but no registry row is a half-built tenant nobody can reach).

5Decision 5 — How you migrate 20 databases at once

This is the part teams forget, and it's the part that hurts most in month six.

Today you have one database and you change it by hand. With database-per-tenant, every schema change must run N times — and if one fails halfway, that customer's database no longer matches your code.

NeededWhat it means
Versioned migrationsAdopt a real tool (Alembic / Sqitch / plain numbered SQL). No more ad-hoc ALTER TABLE by hand. Every change is a file in git.
A schema_version table in every tenant DBSo the runner knows which migrations a given database has already had.
A fan-out runnerA script that reads client_servers, connects to each tenant DB in turn, applies pending migrations, and records success/failure per tenant.
A drift checkA job that compares each tenant's schema against tenant_template.sql and alerts on mismatch. This is how you catch the half-migrated customer before they do.
Backward-compatible changes onlyAdd columns before using them; drop them a release later. You can't migrate all tenants and deploy code in the same instant.
🧩
This is also your on-prem story. Your customer-managed/dedicated deployments are, structurally, just tenant databases you don't host. The same tenant_template.sql + versioned migrations give those clients a supported upgrade path instead of a bespoke one.

7Process — the 11 days, in order

  1. Days 1–2 · Inventory & boundary.
    Classify all 46 tables as central vs tenant. Settle the audit_log question. Output: a one-page table everyone can point at.
  2. Days 3–5 · Redesign the document model.
    Consolidate 4 tables + 4 functions into documents + match_documents. Add the chatbots.id surrogate key. Write the migration for existing data. Highest-value work in the task.
  3. Day 6 · Vector decisions.
    Lock the embedding dimension, the extension (pgvecto.rs vs pgvector), and the index type. Write them into the template.
  4. Day 7 · Credential-storage decision.
    Key Vault vs encrypted columns. Define the column changes so Task 11 has a seam to plug into.
  5. Days 8–9 · Build tenant_template.sql.
    The canonical tenant schema, in git, provisioned via CREATE DATABASE ... TEMPLATE. Prove it by creating two throwaway tenants from it.
  6. Day 10 · Migration process.
    Pick the tool, add schema_version, write the fan-out runner over client_servers, add the drift check.
  7. Day 11 · Review & sign-off with Sahil + Pritam.
    Walk the boundary doc, the template, and the migration runbook. Approval here is what unblocks Task 11.
