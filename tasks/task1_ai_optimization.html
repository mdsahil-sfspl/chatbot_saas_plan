Task 1 — AI pipeline optimization

Two silent issues in your retrieval layer are quietly costing you speed and answer quality: your code uses two different embedding models at two dimensions, and there is no index on the vector column. Fix these first — everything else is built on this foundation.

Where you are today

What the pipeline audit found

bugTwo embedding models — text-embedding-004 (768-d, 26×) and gemini-embedding-001 (3072-d, 6×). Vectors from different models can't be compared.
bugDimension mismatch — search reads EMBEDDING_DIM=3072 while chunk tables are declared vector(768).
slowNo vector index — no hnsw/ivfflat on the embedding column, so every search scans every row.
goodLLM is well-chosen — already on gemini-2.5-flash with streaming in the voice path.
scatterConfig is scattered — model names hardcoded in files, dim in an env var — no single source of truth.
missingNo baseline — no shared way to measure latency & token cost per step, so wins can't be proven.
The headline — fix these two before anything else Your vector search has no index (it scans every document row) and mixes two embedding models at two dimensions (768 vs 3072). So today it both runs slowly and compares vectors that were never meant to be compared — quietly hurting answer quality. Standardising the embedding model and adding an index is roughly 80% of this task's value.

What to do · in parallel, after a shared baseline

The four work-streams

Everyone measures first (Dev D's harness), then each dev optimizes their zone. Dev A's fixes are the priority.

A

Standardize embeddings + index the vectors

the big win
chunking_engine/*admin_router.py (search)api/database_embeddings_change.py
  1. Pick ONE embedding model + dimension. Pragmatic choice: text-embedding-004 at 768-d — most of your pipeline and tables already use it, it's cheap, and it indexes cleanly. (If you want gemini-embedding-001's quality, reduce its output to ≤1536-d, or use halfvec.)
  2. Re-embed every document with that one model/dim in a migration, so nothing old lingers at the wrong dimension. Make the chunk-table vector(...) match exactly.
  3. Add the index — this alone turns a full scan into a fast lookup: 
    -- with one consistent dimension ≤ 2000, a plain hnsw index works
CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops);
-- if you keep > 2000 dims, use halfvec instead of vector
  4. Tune retrieval — pick top_k and the min_similarity threshold against a small set of real questions; confirm the index is used with EXPLAIN.
Why the dimension matters for the indexpgvector's hnsw/ivfflat index supports up to 2000 dimensions for the standard vector type. That's exactly why your 3072-d vectors have no index — they can't have one until you reduce the dimension or switch to halfvec.
B

STT / TTS & voice latency

feel-fast
audio/*audio/stt/* · Sarvam
  1. Stream end-to-end. Feed Sarvam STT and TTS as streams rather than buffering whole audio — this is where conversational lag lives.
  2. Cut round-trips & keep the region close (India), so each audio hop is tens of ms, not hundreds.
  3. Cache repeated TTS (greetings, common replies) so you're not re-synthesising the same phrases.
C

Prompts & process / cost

margins
core/llm_handler.pyLangGraph flow
  1. Trim prompt tokens — shorter templates, only the retrieved context you need — and set a sensible max_output_tokens. Fewer tokens = lower Gemini cost and faster replies.
  2. Cache responses and retrieved context where it's safe to reuse.
  3. Remove redundant LLM calls in the LangGraph flow; keep gemini-2.5-flash for simple turns and only escalate when a turn truly needs it.
D

API / env customization + baseline

foundation
config/settings.pyapi/master_config_receiver.pysystem_settings · app_credentials
  1. One config precedence — define a single order (env → system_settings DB → master-config push) and document it, so keys, models, dims, and temperature all resolve the same way everywhere.
  2. Kill hardcoded model names. Move the embedding model + dimension into that config so they can never drift out of sync again: 
    # one source of truth — not literals scattered across files
EMBEDDING_MODEL = settings.embedding_model   # e.g. "text-embedding-004"
EMBEDDING_DIM   = settings.embedding_dim      # e.g. 768
  3. Build the measurement harness (do this first). Log latency + token cost per step — embed, search, LLM, STT, TTS — so A/B/C can prove their wins instead of guessing.
This is what "customisation of api and env" meansOne clear place decides every model, key, and setting per deployment or tenant — no more values baked into source files. It's also what makes the cloud vs self-host modes (Task on dual-mode) clean.
