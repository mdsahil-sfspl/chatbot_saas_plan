✅
The most important thing first: because you chose database-per-tenant, the scary part of this task — "rewrite every query to add a tenant filter" — mostly disappears. Your data tables don't need a tenant_id column and you don't need row-level security on them. Isolation is physical: each customer's data lives in its own database. Task 11 becomes three focused moves, not a rewrite of your whole data layer.
The picture in plain language
Think of the generic multi-tenant approach as a shared hostel: everyone sleeps in one big room, and you rely on discipline — a name tag (tenant_id) on every bag and a warden (RLS) at the door — so nobody grabs someone else's things. It works, but one slip and a stranger walks off with your bag.

Your Option 3 is separate houses: each customer gets their own house (their own database) with its own lock. There are no name tags on the furniture because nobody else is ever in the house. All you need is a reliable way to walk the right customer to the right front door on every request — and to make sure no back door was left open.

Move A · Tenant-resolution middleware

Where: api/middleware/tenant.py (new file) + register in main.py

What it does: On every request, figure out which customer this is (from the subdomain, e.g. acme.sampurna.app), look them up in clients + client_servers, and stash the answer where the rest of the app can read it — a ContextVar. A ContextVar is request-local: two customers hitting the server at the same time never see each other's value.

# api/middleware/tenant.py  (new)
import contextvars
from fastapi import Request, HTTPException

# request-local slot — safe under concurrency
current_tenant: contextvars.ContextVar = contextvars.ContextVar("current_tenant", default=None)

async def resolve_tenant(request: Request, call_next):
    host = request.headers.get("host", "").split(":")[0]
    subdomain = host.split(".")[0]           # acme.sampurna.app -> "acme"

    # one lookup against the CENTRAL control DB (clients + client_servers)
    tenant = await load_tenant_from_registry(subdomain)
    if tenant is None or tenant["status"] == "suspended":
        raise HTTPException(404, "Unknown or suspended tenant")

    token = current_tenant.set(tenant)      # bind for THIS request only
    try:
        return await call_next(request)
    finally:
        current_tenant.reset(token)          # always release the slot
⚡
Cache the registry lookup. Don't hit the control DB on every single request — wrap load_tenant_from_registry in a small TTL cache (e.g. 30–60s) keyed by subdomain, and bust it when onboarding writes a new row. Otherwise your control DB becomes the bottleneck for every tenant.

Move B · Make get_pool() tenant-aware

Where: core/db_client.py — this is the heart of the change.

Today (single-tenant): there is one global asyncpg pool built from a single DATABASE_URL. Every query in the app goes through it, so everyone would share one database. The fix: keep a dictionary of pools — one per tenant database — and hand back the pool that matches whoever the middleware resolved.

# core/db_client.py  (evolve the existing get_pool)
import asyncpg
from api.middleware.tenant import current_tenant

_pools: dict[str, asyncpg.Pool] = {}     # db_name -> pool, one per house

async def get_pool() -> asyncpg.Pool:
    tenant = current_tenant.get()
    if tenant is None:
        raise RuntimeError("No tenant in context — refusing to open a DB connection")

    key = tenant["db_name"]
    if key not in _pools:
        _pools[key] = await asyncpg.create_pool(
            host=tenant["db_host"], port=tenant["db_port"],
            database=tenant["db_name"], user=tenant["db_user"],
            password=decrypt(tenant["db_password"]),   # see Section 3
            min_size=1, max_size=8,
        )
    return _pools[key]
🎯
This is the whole trick. Because every data query already calls get_pool(), changing this one function re-points your entire data layer at the correct customer database — with zero changes to your query code. That "fail closed if no tenant" guard on line 10 is your safety net: any code path that forgot to set a tenant crashes loudly instead of silently querying the wrong house.
🧮
Watch the pool math. max_size=8 per tenant × N tenants × app workers can blow past Postgres's max_connections. For tens of clients this is fine on separate DB servers. When it grows, cap it — evict idle tenant pools (LRU) or front everything with PgBouncer. This ceiling is one of the reasons Option 3 is a "tens of clients" design, not a "hundreds" design (see Section 5).

Move C · Close the back door — the direct-connect bypass

Where: contextual_chunking.py → _get_conn()

Last session we found this: one module opens its own connection with a raw asyncpg.connect(...) instead of going through get_pool(). In single-tenant that's harmless. In multi-tenant it's the single most dangerous line in the codebase — it's a door into a house that ignores the phonebook, so it can read or write the wrong customer's documents during ingestion.

🚨
Every connection must flow through the tenant-aware get_pool(). Reroute _get_conn() to async with (await get_pool()).acquire() as conn: and delete its private asyncpg.connect. Then add a CI grep that fails the build if asyncpg.connect reappears anywhere outside db_client.py and the onboarding script.
# CI guardrail — add to your test/lint step
# fails if any file (except the two allowed) opens a raw connection
grep -rn "asyncpg.connect" --include="*.py" . \
  | grep -vE "core/db_client.py|onboarding|create_tenant" \
  && echo "BLOCKED: raw connection outside the pool" && exit 1

3The security fix this task exposes

Task 11 makes the registry load-bearing — so the fact that it stores secrets in plaintext becomes urgent, not cosmetic.

🔓
Confirmed in your schema — these columns are plaintext text:
  • client_servers.db_password — the password to every customer database
  • client_servers.api_key and client_api_key_history.api_key
  • client_config.llm_api_key, stt_api_key, tts_api_key — each customer's paid AI keys
Right now, anyone who reads one table (a backup leak, a SQL-injection, a curious admin) gets the keys to every customer's database and AI billing. Under DPDP this is a reportable breach.
WhatWhereHow to fix
DB passwords & API keys in plaintext client_servers, client_config, client_api_key_history Encrypt at rest with an app-held key (Fernet / libsodium), or better store secrets in Azure Key Vault and keep only a reference in the row. The decrypt() call in Move B is where this plugs in.
Onboarding role too powerful for request path per-tenant app connections Request-path connections use a limited Postgres role (read/write data only). CREATE DATABASE lives only in the onboarding routine, with its admin credential in Key Vault.
Verify user passwords aren't plaintext user_credentials.password Column is named password — confirm it holds a bcrypt/argon2 hash, not the raw password. If raw, fix before launch.

4What you deliberately do not do

This is where Option 3 saves you weeks — and where a well-meaning dev could waste them. Say this out loud to the team.

✗ Don't add tenant_id to data tables

No ALTER TABLE ... ADD COLUMN tenant_id on documents, chatbots, conversations. Separate databases already isolate them. Adding it would be dead weight and a false sense of "extra safety."

✗ Don't enable RLS on data tables

Row-level security is the tool for the shared-tables model. With one database per customer there are no other tenants' rows to hide, so RLS policies would guard nothing.

✗ Don't rewrite your queries

You do not touch your 138 endpoints to bolt a tenant filter onto each query. The routing happens once, in get_pool().

✓ Do scope the non-DB stuff

Redis keys, rate-limit keys, and file/upload paths must still be prefixed by tenant (e.g. acme:uploads/...). Those don't get physical separation for free the way the database does.

5Honest pushback — where this design ends

Option 3 is the right call for you now, but be clear-eyed about two things so nobody's surprised in a year:

1. It's a "tens of clients" design, not "hundreds." Every new customer is a new database to migrate, back up, monitor, and connection-pool. At ~10–30 clients that's very manageable. Past ~100, the operational and connection overhead pushes you toward shared tables + tenant_id + RLS — that's the future migration, and it's exactly the "add tenant_id + RLS" work you're skipping today. Track it, don't build it early.

2. The one real failure mode is a routing bug. Physical isolation is strong, but it all hinges on the ContextVar being set correctly on every path. A forgotten tenant, a leaked connection, or a reappearing raw asyncpg.connect can hand one customer another's entire database — worse than a single leaked row. That's why the "fail closed" guard and the CI grep in Section 2 aren't optional polish; they're the safety system for the whole design.

6Process — the order to do it in

  1. Split the control plane out first.
    Confirm the central DB (clients / client_servers / client_config / billing / usage_logs) can be reached independently of any tenant DB. The middleware reads from here.
  2. Encrypt the credentials (Section 3) before wiring routing.
    Do this first so the decrypt() seam exists when get_pool() needs it — and so you never ship a build that reads plaintext keys in the new hot path.
  3. Build Move A (middleware + ContextVar).
    Add the TTL cache for the registry lookup. Test that a bad/suspended subdomain 404s.
  4. Evolve Move B (get_pool() → per-tenant pool cache).
    Include the "no tenant → refuse" guard. Verify two subdomains hit two different databases.
  5. Do Move C (close the _get_conn() bypass) + add the CI grep.
    Nothing merges after this without going through the pool.
  6. Scope Redis / rate-limit / file paths by tenant.
  7. Write the isolation test.
    Seed two tenants, log in as one, assert you cannot read the other's documents/conversations through any endpoint. This is your "done" gate.
