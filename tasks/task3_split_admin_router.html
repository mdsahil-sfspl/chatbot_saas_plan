Task 3 — split admin_router.py

One file holds 4,793 lines and 61 endpoints — billing, users, bots, documents, RBAC, sessions, all tangled together. This task breaks it into ~10 focused routers with zero change to behaviour. The routes already group cleanly, so it's mechanical, not risky.

Where you are today

One file, ten concerns

size4,793 lines · 61 endpoints in a single admin/api/admin_router.py.
mixedTen unrelated domains share the file — Razorpay billing sits next to user CRUD, bots, and RBAC.
frictionMerge-conflict magnet — every dev editing "admin" touches the same file; hard to review, hard to test.
upsideRoutes group cleanly — the paths already cluster by domain (/api/users, /api/bots, /api/payment…), so the seams are obvious.
The golden rule for this task This is a pure mechanical refactor — no behaviour changes. The risk isn't logic, it's accidentally renaming a path or dropping a route. So the whole job is: keep every method + path byte-identical, move one domain at a time, and smoke-test after each. If the route list before equals the route list after, you're done right.
The mental model

One router → ten domain routers

The 61 endpoints fan out into focused files, each owning one concern and one URL prefix.

What to do · in order

The four moves

Extract shared plumbing first, then move domains one at a time, smoke-testing each.

a

Snapshot the routes & extract shared plumbing

~1 day
  1. Capture the exact route list first — this is your safety net. The after-state must match it byte-for-byte. 
    # save the current admin route table to diff against later
python -c "from api.main import app; \
print('\n'.join(sorted(f'{sorted(r.methods)} {r.path}' \
for r in app.routes if hasattr(r,'methods'))))" > routes_before.txt
  2. Pull shared helpers into a common module — the admin auth dependency, DB access, and shared Pydantic models — so the new routers import them instead of duplicating.
  3. Create the package admin/routers/ with an empty router per domain, ready to receive endpoints.
b

Move one domain into its own router

the core loop
  1. Give each router its prefix and tag, then move that domain's endpoints across unchanged. 
    # admin/routers/users.py
from fastapi import APIRouter, Depends
from admin.deps import admin_auth        # shared, from move (a)

router = APIRouter(prefix="/api/users", tags=["users"])

@router.get("")                        # was @router.get("/api/users")
async def list_users(...): ...
  2. Mind the prefix. If you set prefix="/api/users", the decorators become "" and "/{user_id}" — the full path stays identical. (Or keep full paths and no prefix; just be consistent.)
  3. Work smallest-first — portal (2), settings (4), users (5) — to build confidence before the big ones (access 13, bots 9).
c

Wire the routers in & smoke-test each move

after every domain
  1. Include each router where the old one was mounted — same parent, same order. 
    # admin/api/admin_router.py becomes thin wiring
from admin.routers import users, bots, documents, pages, voice_quota, \
                           sessions, access, billing, settings, portal
for m in (users, bots, documents, pages, voice_quota,
          sessions, access, billing, settings, portal):
    admin_app.include_router(m.router)
  2. After each domain moves, boot the app and hit a couple of its routes — don't wait until the end to find a broken import.
d

Verify parity & retire the old file

the proof
  1. Diff the route table. Regenerate the list and compare to your snapshot — an empty diff proves you changed structure, not behaviour. 
    python -c "..." > routes_after.txt
diff routes_before.txt routes_after.txt   # must be empty
  2. Delete the leftover — once every endpoint lives in a domain router, admin_router.py is either thin wiring or gone entirely.
This unblocks other tasksThe billing router you carve out here is exactly where Task 13 (Razorpay subscriptions) plugs in; the users router is where signup's create_user logic gets reused. A clean split pays forward.
