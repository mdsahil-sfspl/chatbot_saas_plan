Task 2 — rate limiting, done right

Your limiter isn't missing — it's stored in the wrong place, and your most expensive paths (chat & voice) run over WebSocket, where it can't reach. Here's exactly what to do, grounded in your code.

Where you are today

What your code does right now

have4 auth endpoints limited — login 3/min, change/forgot/reset_password 1 per 5 min.
flawStored in memory — with WORKERS=4, that's 4 separate counters that don't agree.
gap~102 of 106 endpoints have no limit — including document uploads and vision.
haveUsage quota works — usage_limiter.py caps STT/TTS/OCR seconds per day.
The surprise that changes the plan Your main chat (/ws/chat) and voice (/ws/stt-fast) run over WebSocket. slowapi's @limiter.limit only works on HTTP routes — it cannot limit messages on a socket. So today your most expensive, most abusable paths have zero frequency protection.
The mental model

Two layers, three doors

You guard three kinds of door with two kinds of lock. slowapi (rate = how often) plus your quota (volume = how much). The WebSocket door is the one currently unlocked.

What to do · in order

The three moves

Do them in this order — 2a is the one-line foundation, 2c closes the real gap.

2a

Move the counter into Redis

~half a day · do first
api/session_auth.pyconfig/settings.py
  1. Add the Redis URL to settings (you already run Redis for sessions).
  2. Point the limiter at it — one line. Now all 4 workers share one count instead of four. 
    # api/session_auth.py
limiter = Limiter(key_func=get_remote_address, storage_uri=settings.REDIS_URL)
  3. Verify — hit login 4 times fast; the 4th must be blocked regardless of which worker answers.
Why this comes firstUntil counts are shared, even your good 3/minute login limit is really ~12/minute and resets at random. Every other move depends on this working.
2b

Limit the expensive HTTP endpoints

~1 day
admin/routers/* (uploads)image/router.py (/ask-vision)
  1. Add a key helper so logged-in calls are limited per user, not per shared office IP. 
    # core/ratelimit_keys.py
def user_key(request):
    return getattr(request.state, "user_id", None) or get_remote_address(request)
  2. Cap the cost-heavy routes — document uploads (embedding/OCR) and the HTTP vision endpoint. 
    # uploads — heavy embedding / OCR work
@router.post("/api/bots/upload-documents")
@limiter.limit("10/hour", key_func=user_key)

# image/router.py — vision call
@router.post("/ask-vision")
@limiter.limit("30/minute", key_func=user_key)
  3. Re-key change_password to user_key (it's a logged-in action — IP is the wrong key there).
2c

Limit the WebSocket paths — the real gap

~2–3 days · most important
api/websocket.py (/ws/chat, /ws/stt-fast)core/ws_ratelimit.py (new)
  1. Write a tiny Redis counter — slowapi can't help here, so do it by hand. A few lines. 
    # core/ws_ratelimit.py
async def rate_ok(user_id, bucket, limit, window):
    key = f"rl:{user_id}:{bucket}"
    n = await redis.incr(key)
    if n == 1: await redis.expire(key, window)
    return n <= limit
  2. Throttle messages per user inside the chat / STT handlers (you already read the user from the socket). 
    # api/websocket.py — inside ws_chat, per incoming message
user_id = _get_user_id_from_ws(websocket)
if not await rate_ok(user_id, "chat", limit=60, window=60):
    await ws_safe_send_text(websocket, {"error": "rate_limited"})
    continue   # skip this message, keep the socket
  3. Cap concurrent connections per user at accept time, so one account can't open hundreds of sockets. 
    # at accept
if await redis.scard(f"ws:{user_id}") >= 3:
    await websocket.close(code=1013)  # try again later
  4. Keep the quota in place — voice volume is still capped by usage_limiter.py (seconds/day). 2c adds the missing frequency guard on top.
Why 2c matters mostThis is your only path that touches the LLM and voice spend with no frequency cap today. Closing it is the difference between "limits exist" and "limits protect the expensive stuff."

Definition of done

Done when
  • 2a — the limiter uses Redis storage; the 4 auth limits hold across all 4 workers (verified by a concurrency test).
  • 2b — document uploads and /ask-vision carry limits, keyed by user/tenant; change_password re-keyed to user.
