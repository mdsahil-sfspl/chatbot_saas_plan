Task 10 — refactor WebSocket + session_auth

Two files, 2,591 lines, doing about ten jobs between them. The refactor isn't cosmetic: reading the code, your WebSockets never actually authenticate — they accept the connection and then trust raw cookies as identity. And session_auth.py contains two competing session stores, both with in-memory fallbacks that silently break the moment you run more than one worker.

What the code actually says

Three real problems, not just untidy code

Everything below is read straight out of your two files — this is why the task is P1 and not a nice-to-have.

securityWebSockets trust cookies as identity. ws_stt_fast reads websocket.cookies.get("user_id") and treats it as who you are. A cookie is client-controlled text — anyone can set it.
securityws_chat has no auth at all. It calls websocket.accept() and starts streaming. No token, no session check, no user.
scalingTwo session stores. _MEM_SESSIONS (chat sessions) and _MEM (auth tokens) — separate dicts, separate Redis clients, same file.
scalingIn-memory fallback breaks multi-worker. If REDIS_URL is unset, sessions live in one process's RAM — worker 2 can't see worker 1's login.
sizeTwo giant handlers. ws_chat ≈ 320 lines, ws_stt_fast ≈ 670 lines — each mixing transport, auth, quota, language, LLM, and audio.
goodYou already cap connections. _ws_acquire() + MAX_WS_CONNECTIONS with a 1013 close is genuinely good practice — keep it.
The one that matters mostIn ws_stt_fast, identity comes from websocket.cookies.get("user_id") — and that value is used for voice-quota accounting. Anyone who can open a socket can set that cookie to any value: burn another user's quota, or dodge their own. This is not a refactor detail; it's an open door, and it's why Task 10 sits in P1. Fixing it is the whole point of the task.
Why "runs with T8, T9" and "depends on T3"T3 (splitting the routers) has to land first so you're not refactoring a moving file. Once auth is centralised here, Task 11 gets much easier — the WebSocket tenant-context problem I flagged (sockets skip HTTP middleware) is solved for free, because there'll be exactly one place where a socket establishes identity.
The target shape

One gate in, thin handlers behind it

Today every socket does its own ad-hoc identity guessing from cookies. After the refactor, a single dependency authenticates before the socket is accepted, and the handlers only handle their own job.

1

Authenticate the socket before you accept it

the security fix · do first
api/websocket.pynew: api/ws_auth.py

Right now both sockets accept the connection and then read cookies. Invert it: validate the session token first, close with a policy code if it fails, and only then accept.

Today — websocket.py line 327 & ~676

After — one shared gate (new file: api/ws_auth.py) 
from fastapi import WebSocket
from api.sessions import get_session          # the single store (move 2)

class WSUser:
    def __init__(self, user_id, bot_type, session_id, is_public=False):
        self.user_id, self.bot_type = user_id, bot_type
        self.session_id, self.is_public = session_id, is_public

async def ws_authenticate(websocket: WebSocket) -> WSUser | None:
    """Validate BEFORE accept(). Returns None → caller closes 1008."""
    token = websocket.cookies.get("session_token")     # signed, server-issued
    sess  = get_session(token) if token else None
    if sess:
        return WSUser(sess["user_id"], sess.get("selected_bot", "it"), token)

    # public chatbot path — allowed, but clearly marked as public
    pub = websocket.cookies.get("pub_session_id")
    if pub and await is_valid_public_session(pub):
        return WSUser(f"pub:{pub}", "public", pub, is_public=True)
    return None
Both handlers now start the same way 
@router.websocket("/ws/chat")
async def ws_chat(websocket: WebSocket):
    if not await _ws_acquire():                      # keep your capacity guard
        await websocket.close(code=1013); return

    user = await ws_authenticate(websocket)          # ← BEFORE accept
    if not user:
        await websocket.close(code=1008, reason="unauthorized")
        _ws_release(); return

    await websocket.accept()                          # only now
    # ... user.user_id is TRUSTED from here on
Why "before accept" mattersIf you accept first and check later, you've already spent a connection slot and started the handshake — and every code path after that is one return away from running as an unauthenticated user. Closing with 1008 (policy violation) before accept means the socket never exists for an unauthorized caller.
Keep the public path — just label itYour public chatbots legitimately have no login. Don't break them. The fix isn't "require login everywhere" — it's that a public user gets an explicitly public identity (is_public=True) derived from a server-issued pub_session_id, instead of quietly inheriting whatever user_id cookie the client sent.

Collapse two session stores into one

the scaling fix
api/session_auth.pynew: api/sessions.py

session_auth.py currently holds two independent stores, each with its own Redis client and its own in-memory dict. They don't know about each other.

Today — the two rivals, same file 
# line 61-62 — two separate in-memory dicts
_MEM_SESSIONS: dict[str, str] = {}          # chat session ids
_MEM: dict[str, dict[str, Any]] = {}        # auth tokens

# line 147 — a Redis client built per call
def _redis_client():
    return redis.Redis.from_url(REDIS_URL, decode_responses=True)

# line ~1174 — a DIFFERENT, module-level Redis client
_redis_global = redis.Redis.from_url(_ru, decode_responses=True)
The multi-worker bug hiding in hereBoth stores fall back to a plain Python dict when REDIS_URL is unset. A dict lives in one process's memory. The moment you run multiple uvicorn workers (or a second container), a user who logs in on worker 1 is a stranger to worker 2 — random logouts that are miserable to debug. For a multi-tenant SaaS this must be Redis, always.
After — one store, one client, one TTL policy 
# api/sessions.py (NEW) — the ONLY session store
_redis = redis.Redis.from_url(REDIS_URL, decode_responses=True)

def create_session(user_id, selected_bot="it", tenant_id=None) -> str:
    token = secrets.token_urlsafe(32)
    _redis.hset(f"session:{token}", mapping={
        "user_id": user_id, "selected_bot": selected_bot,
        "tenant_id": tenant_id or "",      # ← Task 11 plugs in here
        "created_at": int(time.time()),
    })
    _redis.expire(f"session:{token}", SESSION_TTL_SEC)
    return token

def get_session(token) -> dict | None:
    return _redis.hgetall(f"session:{token}") or None

def destroy_session(token) -> None:
    _redis.delete(f"session:{token}")
Free win for Task 11Notice tenant_id in the session. Once it's here, the WebSocket tenant problem from Task 11 solves itself: ws_authenticate() returns the user and their tenant, so the socket can call set_tenant() + ensure_tenant_pool() at connect. That's exactly the gap I flagged — this task closes it.
Delete, don't keepapi/session_auth_patch.py is a leftover patch file that imports the old get_session. Once the store is unified, it goes — it's part of the dead-code cleanup you already scoped.

Definition of done

Task 10 is finished when…

Done when
  • No WebSocket calls accept() before authenticating. An unauthorized socket is closed with 1008 and never accepted.
  • websocket.cookies.get("user_id") appears nowhere — identity always comes from a validated session token.
  • Voice quota is billed to the verified user, so a forged cookie can't burn someone else's quota.
  • Public chatbot sockets still work, with an explicitly is_public identity from a server-issued session.
  • There is exactly one session store, backed by Redis; _MEM and _MEM_SESSIONS are deleted.
  • Two workers share sessions — log in against worker 1, make a request served by worker 2, stay logged in.
  • websocket.py and session_auth.py are split; no file over ~400 lines; rate limits preserved.
  • The session carries tenant_id, so sockets can set tenant context — unblocking Task 11's WebSocket gap.
Sequencing adviceShip move 1 (auth before accept) as its own small PR this week, even before the big split. It's the security fix, it's maybe 60 lines, and it's independently reviewable. Moves 2-4 are the refactor and can follow alongside T8/T9.
