Task 7 — error monitoring + logging

Right now, when something breaks on a client's server, you find out because the client tells you — and then you SSH in and scroll through logs hoping to spot it. There's no alert, no history, and no way to tell which client an error came from. This task turns "the client reported a bug" into "we already knew, with the full stack trace, before they noticed."

In plain terms

Think of each client's server as a shop you own in a different city. Today, none of them have a smoke alarm. If a fire starts, you only hear about it when a customer phones to complain — and then you have to drive over and dig through the ashes to work out what happened. This task installs smoke alarms wired to one control room: the moment anything catches fire in any shop, you get a ping that says which shop, what burned, and exactly where — while the customer is still browsing.

What the code shows today

Why you're flying blind

All of this is read straight from your codebase — it's not a criticism, it's the normal state of a product that grew fast. But it's exactly why P1 says "stabilize."

no alertsNothing watches for errors. No Sentry, no monitoring library in requirements.txt. An exception at 2am is invisible until someone complains.
logs vanishLogs only go to stdout. No file, no rotation, no central store. When a container restarts, that history is gone.
no contextLogs can't say which client. No request-id, no tenant-id in log lines — so a multi-client error is a guessing game.
silent failures514 except Exception blocks and ~170 suppress/bare catches. Many swallow the error whole — the bug happens and leaves no trace.
two channels222 print() calls live alongside 766 logger calls. Prints have no level, no timestamp, and can't be filtered or silenced.
leaksRaw errors sent to users. e.g. detail={"error": repr(e)} in main.py — internal exception text (and sometimes DB info) shown to the caller.
The good news — you're closer than it looksYou already call logging.basicConfig() in main.py with a sensible format, and 766 logger. calls already exist across the code. The foundation is there. Task 7 is mostly routing what you already log to somewhere useful, adding the missing context, and putting an alarm on top — not writing logging from scratch.
The target picture

Every error flows to one control room

Two layers. Logging is the diary — everything that happened. Monitoring is the alarm — it only wakes you for the things that matter, with the full story attached.

one log line, two destinations: the alarm sees errors, the diary keeps everything
Why the request-id and tenant-id are the secret sauceA stack trace tells you what broke. The request-id lets you pull every log line for that one request in order — the whole story, not one frame. The tenant-id tells you whose server it was. Without these two, a busy multi-client log is unreadable. With them, you filter to one request from one client in seconds. This is the single highest-value part of the task.
The work · five moves

What to do, where, how

1

Install the alarm (Sentry) — one file, big payoff

do this first · ~1 hour
requirements.txtapi/main.py

Sentry is the smoke alarm. You add one SDK, initialise it once at startup, and from then on every unhandled exception anywhere is automatically captured with its full stack trace and sent to a dashboard that pings you. This is the highest value-per-effort move in the whole task.

requirements.txt — pin it, like your others 
sentry-sdk[fastapi]==2.20.0
api/main.py — initialise before the app is created 
import sentry_sdk
import os

sentry_sdk.init(
    dsn=os.getenv("SENTRY_DSN"),                  # per-deploy, in .env
    environment=os.getenv("DEPLOY_ENV", "prod"),   # prod / staging
    traces_sample_rate=0.1,                          # 10% perf traces
    send_default_pii=False,                        # DPDP — no personal data
)
app = FastAPI(title="Sampurna Voice Agent v2", version="2.0.0")
Cost and fitSentry has a free tier that's plenty for your current scale, and it self-hosts if a client ever demands errors never leave their network. For dedicated clients, each server gets its own SENTRY_DSN in its .env — so errors are already tagged by client, matching your one-config-per-server model.
DPDP note — keep it clean from day onesend_default_pii=False matters. You don't want user messages, emails, or phone numbers flowing into a monitoring tool. Capture the error, not the person. This keeps your compliance surface small.
2

Add request-id + tenant-id to every log line

the readability unlock
new: core/log_context.pyapi/main.py (middleware)

One middleware stamps a unique id on each request and drops both it and the tenant into the log record, so every line automatically carries "who" and "which request" — no code changes in the 766 existing log calls.

core/log_context.py (NEW) — a filter that injects context 
import logging, contextvars

request_id_var = contextvars.ContextVar("request_id", default="-")
tenant_id_var  = contextvars.ContextVar("tenant_id", default="-")

class ContextFilter(logging.Filter):
    def filter(self, record):
        record.request_id = request_id_var.get()
        record.tenant_id  = tenant_id_var.get()
        return True
api/main.py — set the ids per request, add to log format 
logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(name)s "
           "[req=%(request_id)s tenant=%(tenant_id)s] - %(message)s",
)
logging.getLogger().addFilter(ContextFilter())

@app.middleware("http")
async def log_context(request, call_next):
    request_id_var.set(uuid.uuid4().hex[:12])
    tenant_id_var.set(request.session.get("tenant_id", "-"))   # from Task 11
    sentry_sdk.set_tag("tenant", tenant_id_var.get())          # tag errors too
    return await call_next(request)
This dovetails with Task 11 and Task 10The tenant_id comes from the same session context you're building in Task 11, and the WebSocket auth gate from Task 10 lets sockets set it too. The three tasks share one thread of context — build this middleware once and all three benefit.
3

Send logs somewhere that survives a restart

the history fix
docker-compose.yml

Today logs go to stdout and disappear when the container restarts. The good news: because you run Docker Compose, you fix this in the compose file — no code change. Docker captures stdout; you just tell it to keep and rotate the logs.

docker-compose.yml — add a logging block to your app service 
services:
  app:
    restart: unless-stopped        # you already have this — good
    logging:
      driver: "json-file"
      options:
        max-size: "20m"            # rotate at 20MB
        max-file: "5"              # keep 5 files, then recycle
Two tiers, pick per clientSmall / dedicated clients: rotated Docker logs on the box are enough — docker logs plus Sentry for alarms covers you. Cloud SaaS / at scale: ship logs off the box to a central store (Azure Monitor / Log Analytics fits your Azure setup) so you can search across all clients in one place. Start with the compose fix; add shipping when volume justifies it.
4

Stop swallowing errors & stop leaking them

the hygiene pass
api/main.py514 except-blocks (triage)

Two opposite bugs, same root cause — nobody's in charge of what happens when an exception fires. Some blocks eat the error silently; others hand the raw error to the user.

Problem A — silent swallow

the error happens, leaves no trace
try:
    do_thing()
except Exception:
    pass            # ← gone forever

Problem B — raw leak

internal error text shown to caller
except Exception as e:
    raise HTTPException(500,
      detail={"error": repr(e)})  # ← leaks internals
The fix — one global handler, log the truth, return a safe message 
@app.exception_handler(Exception)
async def unhandled(request, exc):
    logger.exception("unhandled error")        # full trace → logs + Sentry
    return JSONResponse(status_code=500, content={
        "ok": False,
        "error": "Something went wrong.",      # safe for the user
        "ref": request_id_var.get(),           # they quote this, you find it
    })
Don't try to fix all 514 at onceThat's a trap. Add the global handler first — it safety-nets everything instantly. Then triage the except blocks in priority order: any that do pass or repr(e) to the user get fixed; the rest can wait. Give the user a reference id, not a stack trace — they read you the ref, you search the log, done.
Rule of thumb for the teamAn except block may swallow an error only if it also logs it. "Catch and continue" is fine; "catch and vanish" is a future 2am mystery. Make logger.exception(...) inside every catch the team standard.
5

Retire the 222 print() calls

the cleanup · lowest priority
across the repo

print() and logger are two mouths talking over each other. Prints have no level, no timestamp, no request-id, and can't be turned down in production. Convert them so everything flows through the one logging pipeline you just built.

# before
print(f"user {uid} did the thing")
# after
logger.info("user %s did the thing", uid)     # now has level, time, req-id, tenant
This one is a background chore, not a blockerDo it opportunistically — whenever someone touches a file with a print(), they convert it. A repo-wide search-and-replace is risky; a steady "boy-scout rule" cleanup is safe. It's listed last on purpose: moves 1-4 give you the monitoring; this just tidies the pipe feeding it.
