Task 9 — CI/CD pipeline setup

Today, shipping means someone SSHs into each client's VM and runs git pull + docker compose up --build -d by hand. It works — until you have five clients, someone forgets one, or a broken commit reaches production because nothing checked it first. This task builds the automatic gate: every push gets tested and built before it can reach a client, and deploying stops being a person remembering to do things in order.

In plain terms

Right now your kitchen sends food straight to the customer — nobody tastes it first. Most of the time it's fine, because your cooks are careful. But one bad dish goes out and the customer finds it before you do. CI/CD is the tasting station: every dish stops there, gets checked automatically, and only then goes to the table. And because the same checked dish goes to every branch of the restaurant, no branch gets a different version by accident.

The uncomfortable part of Task 9

A pipeline with no tests is a pipeline that only checks spellingYour task board says "CI/CD pipeline setup" — but the pipeline's whole job is to stop bad code, and right now there is nothing for it to run. Both test_*.py files are empty stubs. If you build the pipeline today, it will confirm your code imports and your image builds. That's real value — but it will not catch a broken login, a leaked tenant, or a dead endpoint. It will give you a reassuring green tick that means much less than it looks like it means.
So do it in this orderBuild the pipeline anyway — it's cheap and the build/security gates pay for themselves immediately. But treat "add a first handful of real tests" as part of this task, not a separate future one. You don't need 500 tests. You need five: does the app start, does /health answer, does login reject a bad password, does an unauthenticated request get refused, does a tenant-scoped query stay in its tenant. Those five turn the green tick into a real gate.
Why the board is right that T9 depends on T7Automated deploys mean code reaches production without a human watching. That's only safe if something else is watching — which is Task 7's monitoring. Ship the alarm first, then let the pipeline pull the trigger. Automating deploys before you can see failures is how a small bug becomes a silent outage.

The work · five moves

What to do, where, how

1

The first workflow — build + import check

start here · ~2 hours
new: .github/workflows/ci.yml

Before tests, before anything clever — get a workflow running that proves the image builds and the app imports. This alone catches the most common breakage: someone adds an import and forgets the dependency.

.github/workflows/ci.yml (NEW) 
name: CI
on:
  push:
    branches: [main, develop]
  pull_request:

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build the Docker image
        run: docker build -t sampurna:ci .        # same Dockerfile as prod

      - name: App imports cleanly
        run: docker run --rm sampurna:ci python -c "import api.main"
Why this catches your specific riskYour Dockerfile uses pip install --no-deps, which installs exactly what's listed and resolves nothing. If a transitive dependency is missing from the freeze, the container builds fine and then crashes at runtime — on a client's server. The import api.main step catches that in CI instead, for free.
2

Write the first five tests

the part that makes CI mean something
new: tests/delete: api/test_search_function.py

Not a test suite — a smoke test. Five tests that would each catch a real outage. Put them in a proper tests/ folder and delete the two stub files so nothing is pretending to be a test.

tests/test_smoke.py (NEW) 
from fastapi.testclient import TestClient
from api.main import app

client = TestClient(app)

def test_app_starts():
    assert app is not None                      # imports + wiring OK

def test_health_endpoint():
    assert client.get("/health").status_code == 200

def test_login_rejects_bad_password():
    r = client.post("/login", json={"user_id": "x", "password": "wrong"})
    assert r.status_code in (401, 403)

def test_protected_route_needs_auth():
    assert client.get("/me").status_code in (401, 403)
The fifth test comes from Task 11Once tenant routing exists, add test_fail_closed — a query with no tenant in context must raise, never fall back to a default database. That single test protects the most expensive possible bug in a multi-client SaaS, and it belongs in the pipeline from the day Task 11 lands.
Resist the urge to chase coverageDon't set a coverage target. With 138 endpoints you'd never hit it and the team would start writing meaningless tests to satisfy a number. Five tests that each map to a real outage beat 200 that assert nothing.
3

Add the quality + security gates

tools you already have
.github/workflows/ci.yml

pytest, bandit and mypy are already in your requirements.txt. Wire them into the workflow so they run on every push instead of only when someone remembers.

      - name: Tests
        run: docker run --rm sampurna:ci pytest -q tests/

      - name: Security scan (bandit)
        run: docker run --rm sampurna:ci bandit -r api core -ll   # high+med only

      - name: Dependency audit
        run: pip install pip-audit && pip-audit -r requirements.txt

      - name: Secret scan
        uses: gitleaks/gitleaks-action@v2      # no keys committed, ever
Start the strict tools in "warn" modemypy on a large existing codebase will produce hundreds of complaints on day one. Add it as non-blocking first (continue-on-error: true), fix gradually, then make it a hard gate. A gate everyone routinely overrides is worse than no gate — it teaches the team that red means nothing.
The secret scan earns its placeYour .gitignore and .dockerignore already exclude .env — good. A secret scanner is the backstop for the day someone pastes a key into a config file instead. Given your control tables currently hold plaintext credentials, this is cheap insurance.
4

Automate the deploy — carefully

the CD half
new: .github/workflows/deploy.ymlnew: deploy.sh on each VM

Your current deploy — git pull + docker compose up --build -d — becomes a script the pipeline runs, with a health check and a rollback. Same commands, just no longer dependent on a person doing them right.

deploy.sh — lives on each client VM 
#!/usr/bin/env bash
set -euo pipefail                          # stop on first error

PREV=$(git rev-parse HEAD)                 # remember where we were
git pull --ff-only
docker compose up --build -d

# wait for the app to actually answer
for i in {1..30}; do
  if curl -fsS http://localhost:8000/health >/dev/null; then
    echo "✅ healthy"; exit 0
  fi
  sleep 2
done

echo "❌ health check failed — rolling back"       # ← the safety net
git reset --hard "$PREV"
docker compose up --build -d
exit 1
.github/workflows/deploy.yml — runs after CI passes 
name: Deploy
on:
  workflow_run:
    workflows: [CI]
    types: [completed]
    branches: [main]

jobs:
  deploy:
    if: github.event.workflow_run.conclusion == 'success'
    runs-on: ubuntu-latest
    environment: production          # ← requires manual approval
    steps:
      - name: Deploy to client VM
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.VM_HOST }}
          username: ${{ secrets.VM_USER }}
          key: ${{ secrets.VM_SSH_KEY }}
          script: cd /opt/sampurna && ./deploy.sh
Do not auto-deploy to client servers on day oneStart with manual approval (the environment: production line does this — GitHub will wait for a human to click Approve). Fully automatic deploys to paying clients' servers, with no tests worth the name and monitoring only just installed, is how you find out about a bad release from an angry customer. Earn the automation: first the tests, then the alarm, then remove the approval step.
Where a staging environment fitsThe cheapest useful version: one small VM running the same compose stack against a throwaway database. The pipeline deploys there automatically with no approval, you glance at it, and only then does the production job ask for its click. It catches the "works in CI, dies on a real server" class of bug — which is exactly the class that CI can't catch by design.
5

Make the repo enforce the gates

config, not code · ~30 min
GitHub repo settings

A pipeline nobody has to obey isn't a gate. Two settings in GitHub turn the checks from advisory into mandatory.

  1. Protect main. Require the CI workflow to pass before merge, and require at least one review. With 3-4 devs this is light — but it means broken code physically cannot land on the branch that deploys.
  2. Tag every release. Have the pipeline tag each successful build with the commit SHA. When a client says "it broke this morning," you can name the exact release and roll back to a known-good one instead of guessing.
  3. Store every secret in GitHub Secrets — SSH keys, VM hosts, the Sentry DSN. Never in the workflow file, which is committed to the repo like any other code.
One rule worth agreeing as a teamIf the pipeline goes red, fixing it is the priority over new work. The failure mode for small teams isn't the absence of CI — it's a CI that's been red for three weeks and everyone has learned to ignore. A red pipeline that nobody trusts is worse than no pipeline, because it gives you false confidence at exactly the wrong moment.

Definition of done

Task 9 is finished when…

Done when
  • Every push and PR runs a workflow that builds the real Dockerfile and confirms the app imports.
  • There is a genuine tests/ folder with at least five meaningful tests; both stub files are deleted.
  • Tests, bandit, dependency audit and secret scanning all run automatically on every push.
  • main is branch-protected — code that fails the pipeline cannot be merged.
  • Deploying is one action, not a checklist: the pipeline runs deploy.sh, which health-checks /health and rolls back automatically if the app doesn't come up.
  • Production deploys require a human approval click (until tests and monitoring have earned full automation).
  • Every deploy is tagged, so any release can be identified and rolled back to by name.
  • All secrets live in GitHub Secrets — never in a committed workflow file.
Sequencing adviceDo move 1 this week — a workflow that builds your image and imports the app is two hours and immediately catches the --no-deps class of bug. Then move 2 (the five tests), because everything else in this task is only as valuable as what the pipeline actually runs. Moves 4 and 5 wait until Task 7's monitoring is live — automating deploys before you can see failures is the one genuinely risky ordering here.
