Security review — what's exposed, why, and the fix

An honest read of the platform's security posture. Each issue below is grounded in your actual files, with what it is, why it becomes a problem, and what to do. The good news: several foundations are already strong — the gaps are specific and fixable.

Posture at a glance

Strong foundations, specific gaps

Already strong — keep it

  • Passwords hashed with Argon2 (modern, correct choice)
  • Strong security headers — CSP, HSTS, X-Frame, nosniff, COOP/COEP
  • Master-portal pushes are Fernet-encrypted in transit
  • Master surface is key-guarded (X-Master-API-Key on every call)
  • Login / reset endpoints are correctly public and rate-limited

Needs work — the gaps

  • Credentials committed to the git repo & history
  • A single master key is a skeleton key to all admin ops
  • Proxy trust is open — client IPs can be spoofed
  • Only 4 of 138 endpoints are rate-limited
  • CORS can fail open; no error monitoring to catch abuse

The headline: your endpoint count isn't the risk — the inconsistency is. Some doors are well locked; a few are not, and one very powerful key guards a duplicated surface.

The issues · worst first

What's wrong, why, and the fix

Critical

Credentials committed to the repository

#1
tools/database_data/user_credentials_rows.csvpassword_reset_tokens_rows.csv
What it is

Two CSV exports containing real user credentials and password-reset tokens are checked into the repo — and therefore into its git history.

Why it becomes a problem

Anyone who ever clones the repo — a contractor, a leaked laptop, a mis-set repo visibility — gets those secrets. Because they're in history, deleting the file today does not remove them; the old commit still holds them. If master_api_key was ever in a committed .env, that skeleton key (issue #2) may be exposed the same way.

What to do — in your codebase

Rotate first — treat those credentials and tokens as compromised and reset them. Then remove the files and scrub git history, and block the folder so it can't recur.

git rm --cached tools/database_data/user_credentials_rows.csv \
              tools/database_data/password_reset_tokens_rows.csv
echo "tools/database_data/" >> .gitignore
git filter-repo --path tools/database_data/user_credentials_rows.csv --invert-paths
High

One master key is a skeleton key to all admin operations

#2
api/master_admin_bridge.py (_verify_master_key)system_settings.master_api_key · .env MASTER_API_KEY
What it is

The master bridge re-exposes ~22 admin operations (users, bots, quota, sessions, settings, reports) at /api/master/admin/*, all guarded by a single static X-Master-API-Key header.

Why it becomes a problem

That one key can create users, delete bots, and change settings — the full admin surface — bypassing the session-cookie login entirely. So your admin security now rests on how that single key is generated, stored, rotated, and kept out of logs. One leak = total admin compromise. This is the concrete cost of the duplicated surface: two doors to the same room, and the second door's lock is a master key.

What to do — in your codebase

Treat master_api_key as your highest-value secret. Generate it long & random, keep it out of the repo, and rotate it regularly. Add audit logging on every bridge call (who, what, when). Longer term, move from one shared key to per-caller keys or mTLS, and make the bridge delegate to the same admin service functions rather than duplicating logic — so there's one place to secure, not two.

High

Proxy trust is open — client IPs can be spoofed

#3
api/main.py:128 — ProxyHeadersMiddleware(trusted_hosts="*")
What it is

The app trusts the X-Forwarded-For header from any client, not just your load balancer.

Why it becomes a problem

An attacker can forge that header to pretend to be any IP. That silently defeats every IP-based rate limit (issue #4) and poisons your logs and audit trails with fake source IPs — so even the little abuse protection you have becomes bypassable, and forensic data becomes untrustworthy.

What to do — in your codebase

Trust proxy headers only from your load balancer's IP, from an env var — never "*".

TRUSTED = os.getenv("TRUSTED_PROXY_IPS","").split(",")
app.add_middleware(ProxyHeadersMiddleware, trusted_hosts=TRUSTED or "127.0.0.1")
High

Almost nothing is rate-limited

#4
api/session_auth.py:26 — in-memory Limiter4 of 138 endpoints
What it is

Only 4 endpoints (login + password flows) have a rate limit, and the limiter is in-memory — so it isn't even shared across your 4 workers. The chat/voice WebSockets and every admin write have no throttle at all.

Why it becomes a problem

Unthrottled endpoints invite brute-force, scraping, and cost-abuse — someone can hammer the expensive AI/voice paths, or enumerate data through unprotected admin writes. And because the limiter is per-worker in-memory, a "3/min" limit is really "3 × workers." Combined with issue #3, even those limits are bypassable.

What to do — in your codebase

Back the limiter with Redis (you already run Redis) so limits are shared, then extend coverage to the expensive AI endpoints and add manual per-connection limits on the WebSockets. Key by user/tenant after login, by (fixed) IP before.

Limiter(key_func=get_remote_address,
        storage_uri="redis://…")   # shared across workers
Medium

CORS can fail open

#5
api/main.py — CORS fallback
What it is

If ALLOWED_ORIGINS is unset, CORS falls back to ["*"] — together with allow_credentials=True.

Why it becomes a problem

Wildcard origins plus credentials is the classic misconfiguration that lets any website make logged-in requests to your API on a user's behalf (cross-site data theft). It only takes one deployment where that env var is forgotten to open the door.

What to do — in your codebase

Fail closed: require an explicit origin list and refuse to start without it — never fall back to "*".

origins = [o for o in os.getenv("ALLOWED_ORIGINS","").split(",") if o]
if not origins: raise RuntimeError("ALLOWED_ORIGINS must be set")
Medium

No error monitoring — attacks & failures are invisible

#6
no Sentry · 222 print() · 514 broad except
What it is

There's no error monitor, logging is ad-hoc (222 print()s), and 514 broad except Exception blocks can swallow failures silently.

Why it becomes a problem

Security matters here too: without monitoring you can't see an attack in progress — a spike of failed logins, a flood on the AI endpoints, repeated 500s from a probe. Broad excepts that swallow errors mean an exploited failure can run indefinitely with no signal to anyone.

What to do — in your codebase

Add Sentry with a global exception handler so every unhandled error is captured and alerts the team; make the broad excepts log with context instead of swallowing; add a request ID so one suspicious session is traceable.

Verify

Confirm admin routes enforce auth globally

#7
admin/api/admin_router.py
What it is

A per-endpoint scan couldn't see auth on ~20 admin write-routes. This is likely because admin auth is applied at the router/middleware level (which the scan can't detect), not a real hole — but it's unconfirmed.

Why it becomes a problem

If any of those 61 admin routes is not covered by a global admin-auth dependency, it would be an unauthenticated admin action — the most severe kind of gap. The uncertainty itself is the risk.

What to do — in your codebase

Confirm the admin router (or its parent app) applies a global admin-auth dependency so every route inherits it. If it relies on per-endpoint checks, add the missing ones. This is a 10-minute read that turns a "maybe" into a "no."

Recommended sequence

Fix in this order

Priority order
  1. Rotate & purge the committed credentials (#1) — and rotate master_api_key while you're at it. Nothing else matters if secrets are already out.
  2. Verify admin routes are globally authed (#7) — cheap, and rules out the worst-case.
  3. Lock proxy trust (#3) so rate limits and logs become trustworthy.
  4. Add real rate limiting (#4, Redis-backed) across auth, AI, and WebSocket paths.
  5. Fail CORS closed (#5) and harden the master_api_key handling / add bridge audit logging (#2).
  6. Add Sentry + a global handler (#6) so you can see the next problem before your users do.

Every item here already maps to a task in your stabilize phase — so securing the platform isn't extra work, it's your existing plan, sequenced by risk.
